# 0002: Wesley Schema Profile

**Status:** DESIGN
**Date:** 2026-03-26

## Purpose

This note defines the compiler-first stance for using GraphQL and Wesley in the
WARP TTD ecosystem.

The goal is not "GraphQL everywhere."

The goal is:

- GraphQL syntax
- Wesley semantics
- deterministic generated artifacts

## The Rule

Raw GraphQL semantics are **not** the source of truth.

A Wesley-controlled subset/profile/dialect is the source of truth.

That profile exists to support:

- deterministic code generation
- stable versioning
- deterministic serialization
- rewrite contract declaration
- footprint declaration
- cross-host protocol compatibility

## Why This Matters

Ordinary GraphQL ecosystems tend to pull toward:

- resolver-driven behavior
- dynamic server semantics
- CRUD-shaped thinking
- incidental nondeterminism
- runtime magic

That is poison for WARP.

WARP needs something closer to:

- a declarative IR
- a typed operation language
- a deterministic codec authority
- a versioned schema contract
- a compiler-owned safety surface

## Current Evidence

This direction is not speculative from zero.

Observed building blocks already exist:

1. Wesley already treats GraphQL as a human-authored source of truth and emits
   generated artifacts from it.
2. Wesley already defines directives such as:
   - `@wes_footprint`
   - `@wes_codec`
   - `@wes_version`
3. Echo already has runtime schema fragments authored in GraphQL.
4. Echo already has a vendored TTD manifest/IR staging area generated by
   Wesley-facing tooling.

So the design move is to harden and narrow an existing direction, not invent a
totally unrelated architecture.

## Allowed Responsibilities

The Wesley profile may define:

1. type shapes
2. operation shapes
3. explicit footprint declarations
4. codec metadata
5. version metadata
6. protocol envelopes
7. manifest/IR generation rules

## Not Allowed

The trusted core must not depend on:

1. arbitrary userland resolver behavior
2. host-specific runtime callbacks masquerading as schema semantics
3. library-defined serialization behavior that changes underneath us
4. open-ended GraphQL feature creep
5. nondeterministic execution hidden behind convenience abstractions

## Proposed Allowed Surface

The first Wesley profile for WARP TTD should stay narrow.

### Allowed type constructs

1. object types
2. enums
3. scalars with Wesley-owned semantics
4. input objects
5. lists and optionals where codegen behavior is deterministic
6. directives explicitly recognized by Wesley

### Allowed operation constructs

1. explicitly declared read-only query/result envelopes for TTD
2. explicitly declared mutation/rewrite envelopes where needed
3. operation metadata required for versioning, codec generation, and footprint
   declaration

### Allowed directive families

1. footprint directives
2. codec directives
3. version directives
4. registry/identity directives
5. additional WARP-specific directives only if Wesley owns their semantics

## Banned Or Deferred Features

Until proven otherwise, treat these as banned or outside the trusted profile:

1. arbitrary resolver-side semantics
2. implicit execution semantics derived from generic GraphQL tooling
3. subscriptions as a core semantic dependency
4. federation-style cross-service composition semantics
5. custom semantics that Wesley cannot validate or encode deterministically

## Serialization Rule

Serialization behavior must be compiler-defined, not library-defined.

That means:

- canonical encoding rules are owned by Wesley/codegen
- schema version changes are explicit
- generated Rust/TS/JS artifacts must agree byte-for-byte where applicable

## Footprint Rule

Footprints are first-class Wesley concepts, not comments.

That implies:

1. an operation may declare a footprint in schema
2. Wesley should validate required footprint presence and shape
3. generated artifacts should expose footprint metadata to hosts and tools
4. runtime checks may still exist, but omission/drift should fail as early as
   possible

## Consequence

The right framing is:

Wesley is not being used to "add GraphQL."

Wesley is being used to define a shared, deterministic, versioned graph
operation language whose syntax happens to borrow from GraphQL.
